字符串String StringBuffer StringBuilder 对比

Veröffentlicht am 2017-02-02 | in java | Visitors

区别

java 中常用的三种字符串类为String 、StringBuffer 和StringBuilder。总结三者的区别为：

  • String:字符串常量。String 是不可变的对象，每次对String 类型进行改变时相当于创建了一个新的String对象,原来的变量名指向新的对象

  • StringBuffer：线程安全的字符串变量。在内存中保存的都是一个有序的字符串序列（char 类型的数组），StringBuffer对象的值是可以改变的。

  • StringBuilder：线程不安全的字符串变量。和StringBuffer类基本相同，StringBuilder对象的值也是可以改变的，主要区别是StringBuilder是线程不安全的

使用场景

  • String：在字符串不经常变化的场景中可以使用String类，例如常量的声明，少量的变量运算
  • StringBuffer：在频繁进行字符串运算（如拼接、替换、删除等），并且运行在多线程环境中。例如XML解析、HTTP参数解析和封装。
  • StringBuilder：在频繁进行字符串运算（如拼接、替换、删除等），并且在单线程的环境中，在不出现多线程的问题时常常使用StringBuilder代替StringBuffer。

性能分析

在字符串变量拼接时的效率StringBuilder > StringBuffer>String。

原因：

  • String在进行字符串拼接修改时，由于String是不可变字符串，所以在实际过程中是产生一系列对象，这样对系统性能产生影响。引用变量会指向最后一个对象，中间这些无用对象被存在堆内存中，JVM的垃圾回收就会工作，所以性能最差。
  • StringBuffer和StringBuilder只是一个字符数组的扩容而已，每次修改都会对StringBuffer对象本身进行操作，而不是产生新的对象，所以性能比较好。
  • StringBuffer是线程安全的，加了同步锁，所以相比于StringBuilder非线程安全的效率低些。

特殊情况

在某些String对象的字符串拼接其实被JVM解释成了StringBuffer对象的拼接，所以效率不会比StringBuffer拼接慢,下面这种情况下更快：

生成str对象的速度远大于sbf,这是因为在jvm运行的过程中，

String str = "This is "+"a"+"java"+"testing";其实就是String str ="This is a java testing";所以速度较快。

equal 和 ==

==

用于比较两个对象的时候，是用来判断两个引用是否指向了一块内存。

特殊情况：

这是因为这两种创建方法是有差别的，str4和str5的在常量池中创建，常量池是为了避免频繁的创建和销毁对象而影响系统性能，实现了对象的共享。在常量池中所有相同的字符串常量被合并只占用一个空间，str5实际拿的是常量池中拿对象，所以在进行比较字符串时，此时==其实比较的就是实际的内容是否相等。如果采用new的方式则是在堆内存中创建一个新的对象。

equals()

equals()是object的方法，默认情况下，和==一样，比较的是引用变量的地址。但是String类中的equals()方法进行重载，比较的是对象的value值
